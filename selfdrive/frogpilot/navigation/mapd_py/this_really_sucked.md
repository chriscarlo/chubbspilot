# Summary of mapd_py SCons Build Debugging

This document summarizes the steps taken to resolve build errors when integrating the `selfdrive/frogpilot/navigation/mapd_py` Cython module into the main openpilot SCons build process on a comma 3x device using pyenv.

**Initial Goal:** Compile the `mapd_py` module using the existing SCons setup.

**Problems Encountered & Solutions:**

1.  **`ModuleNotFoundError: No module named 'Cython'`**
    *   **Problem:** The `python3` executed by the SCons `env.Command` for `setup_capnp.py` was not using the correct pyenv environment, despite `which python3` showing the correct shim.
    *   **Solution:** Modified `selfdrive/frogpilot/navigation/mapd_py/SConscript` to use the *full absolute path* to the pyenv python executable (e.g., `/usr/local/pyenv/versions/3.11.4/bin/python3`) in the `build_command_str`.

2.  **`ValueError: '*_capnp_cython.pyx' doesn't match any files`**
    *   **Problem:** The `capnp compile ... -o<capnpc-cython_plugin>:<output_dir>` command, when run by SCons from the root, was incorrectly creating the `.pyx` file in a nested directory structure (e.g., `mapd_py/selfdrive/frogpilot/navigation/mapd_py/`) instead of the specified `output_dir` (`mapd_py/.`). The `--src-prefix=.` flag did not resolve this for the cython plugin.
    *   **Solution:** Modified the `cython_command_str` in `SConscript` to first `cd` into the target directory and then run `capnp compile` with relative input (`offline.capnp`) and output (`.`) paths.

3.  **`fatal error: offline.capnp.h: No such file or directory`**
    *   **Problem:** The C++ compiler, invoked by `setup_capnp.py` during the Cython extension build, couldn't find the `offline.capnp.h` header file generated by the C++ Cap'n Proto step.
    *   **Solution:** Modified `selfdrive/frogpilot/navigation/mapd_py/setup_capnp.py` to explicitly add the current directory (`.`) to the `include_dirs` list for the Cython `Extension` object before passing it to `setup()`.

4.  **`RuntimeError: You need to run `capnp compile -oc++` in addition to `-ocython` first.`**
    *   **Problem 1:** Initially suspected SCons dependency ordering issues (parallelism or incorrect dependency declaration). The `setup_capnp.py` script ran before the `capnpc -oc++` command had created `offline.capnp.c++`.
    *   **Solution 1:** Added explicit `env.Depends()` calls in `SConscript` to force the C++ generation (`cpp_command`) to complete before the Cython build step (`build_command`). This did *not* solve the underlying issue.
    *   **Problem 2 (Root Cause):** The `capnpc -oc++` command itself was failing silently when invoked by SCons with the original path structure (`-oc++:selfdrive/frogpilot/navigation/mapd_py`). Manual testing confirmed it only worked correctly when run *after* changing into the target directory.
    *   **Solution 2:** Modified the `cpp_command_str` in `SConscript` (similar to the Cython command fix) to first `cd` into the target directory and then run `capnpc -oc++` with a relative input path (`offline.capnp`) and default output path (`.`).

**Final State:** The build completes successfully, generating the required `.so` file. Some compiler warnings related to unused Cap'n Proto variables remain but are likely harmless.
